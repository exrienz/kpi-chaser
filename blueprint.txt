# 1. Clarifying Questions

Before implementation, a few design decisions should be confirmed:

1. Should **AI requests run synchronously** or via **background workers**?
2. Should users be able to **edit AI enhanced text** before saving?
3. Should KPI mapping be **AI only** or allow **manual override**?
4. Will attachments (screenshots) be part of **MVP**?
5. Should the system support **multiple AI providers** or just **OpenRouter-compatible APIs**?

For this blueprint I assume **AI via OpenRouter but provider-agnostic**.

---

# 2. Assumptions

Assumptions for MVP:

* Self-hosted deployment
* Multi-user (small team or personal instance)
* Email + password authentication
* No external integrations
* AI provider configurable (OpenRouter compatible)
* Storage requirements small
* AI calls asynchronous
* SQLite database (edge optimized)
* Screenshot logging optional but supported

# 3. System Architecture Overview

We use a **modular monolith**.

Why:

* Simpler deployment
* Lower memory footprint
* Easy migration to microservices later

Main components:

### Web UI

Simple SPA.

Responsibilities:

* logging achievements
* viewing KPI progress
* generating reports

### API Server (Go)

Handles:

* auth
* KPI management
* achievement logging
* AI requests
* reporting

### Worker

Background jobs:

* AI rewriting
* KPI classification
* report generation

### SQLite Database

Stores:

* users
* kpis
* achievements
* ai outputs
* reports

### AI Provider Layer

Adapter supporting:

* OpenRouter
* OpenAI-compatible APIs

# 8. Concurrency Strategy (Go)

Workers implemented with **goroutines + channels**.

Example worker pool:

```
jobs channel
     ﾃ｢魹ｽ魹ｽ
 ﾃ｢魹ｽ魹ｽﾃ｢魹ｽ魹ｽﾃ｢魹ｽ魹ｽﾃ｢魹ｽ魹ｽﾃ｢魹ｽﾂｼﾃ｢魹ｽ魹ｽﾃ｢魹ｽ魹ｽﾃ｢魹ｽ魹ｽﾃ｢魹ｽ魹ｽﾃ｢魹ｽ魹ｽ
 ﾃ｢魹ｽﾂｼ   ﾃ｢魹ｽﾂｼ    ﾃ｢魹ｽﾂｼ
W1  W2   W3
```

Pseudo code:

```go
func StartWorkers(n int, jobs <-chan Job) {
    for i := 0; i < n; i++ {
        go worker(jobs)
    }
}
```

Backpressure:

```
queue size limited
```

Rate limiting for AI calls:

```
golang.org/x/time/rate
```

# 10. Performance Strategy

Designed for **low resource systems**.

Key decisions:

### SQLite WAL mode

```

```

### AI jobs async

Prevents UI blocking.

### Worker pool

Prevents CPU spikes.

### DB connection pool

```
max_open = 5
```

Small API responses.
